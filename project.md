# Проект «Morf»

## 1. Цель и описание

Создание сообщества художников для еженедельных тренировок по созданию персонажей.

Каждую неделю администратор формирует набор картинок-референсов, описывающих свойства персонажа (например: «окружение», «стиль», «руки», «вселенная»). Участники сообщества рисуют своих персонажей на основе текущего набора референсов и публикуют свои работы в общую ленту. Другие пользователи могут просматривать ленту и ставить лайки.

---

## 2. Пользовательские сценарии

### 2.1 Анонимная регистрация и авторизация
- Регистрация полностью анонимна, без сбора email, телефона или реальных имени/фамилии.
- При первом входе пользователь придумывает публичный nickname.
- Сразу после этого на устройстве генерируется токен, который хранится в `localStorage`.
- Бэкенд хранит только SHA-256 хеш этого токена (`auth_hash`) — именно по нему происходит идентификация пользователя при каждом запросе.
- Для смены устройства предусмотрен одноразовый recovery code. Пользователь может получить его в настройках, а затем активировать на новом устройстве, сохранив свой аккаунт и историю работ.

### 2.2 Лента работ
- Глобальная лента всех опубликованных работ (с пагинацией).
- Доступен фильтр по конкретному набору референсов (неделе).
- Гость (неавторизованный пользователь) может просматривать ленту, но не может ставить лайки или загружать работы.

### 2.3 Публикация работы
- Авторизованный пользователь загружает изображение своей работы.
- Работа привязывается к конкретному набору референсов (неделе).
- Можно добавить подпись (caption) к работе.
- После загрузки работа сразу появляется в ленте (постмодерация).

### 2.4 Лайки
- Авторизованные пользователи могут ставить и убирать лайки на чужих работах.
- Повторный лайк с того же аккаунта на ту же работу невозможен.
- Счетчик лайков денормализован и хранится в таблице `artworks` для быстрой выдачи ленты.

### 2.5 Комментарии
- В схеме БД таблица `comments` заложена как заготовка на будущее.
- В MVP (первой версии) API для комментариев не реализуется.

## 3. Администрирование

### 3.1 Управление референсами
- Администратор создает категории референсов (свойства персонажа) с произвольными slug и порядком сортировки.
- Администратор загружает изображения-референсы (с метаданными: разрешение, размер, MIME-тип) в хранилище (CDN).
- Администратор формирует набор (`reference_set`) на неделю, выбирая произвольное количество картинок из разных категорий.

### 3.2 Автопубликация наборов
- Администратор может либо вручную публиковать набор каждую неделю, либо создать несколько наборов заранее с указанием даты публикации (`published_at`).
- Система автоматически делает набор активным, когда наступает его дата публикации.

### 3.3 Модерация работ
- **Постмодерация:** работы видны в ленте сразу после загрузки.
- Администратор может одобрить (`approved`), отклонить (`rejected`) или скрыть (`deleted` via soft delete) работу.
- Все действия модерации логируются в таблице `moderation_actions` (аудит).
- У отклоненных работ статус меняется, но запись остается в базе.

### 3.4 Админ-панель
- Для управления контентом используется простая админ-панель «из коробки» (типа Laravel Nova, Filament или аналогичного готового решения), интегрированная в проект.
- Через нее администратор управляет категориями, референсами, наборами, модерацией работ и статическим контентом.

### 3.5 Статический контент
- Администратор может редактировать текст приветствия для новых пользователей.
- Администратор может добавлять/изменять ссылки на донаты и социальные сети проекта.
- Эти данные хранятся в таблице `project_info` в формате key-value (value может быть JSON).

## 4. Технические требования

### 4.1 Архитектура
- **Backend:** Laravel (последняя стабильная версия).
- **Архитектурный стиль:** слоистая / чистая архитектура с явным разделением на Bounded Contexts (Identity, Content, Artworks, Engagement, Moderation, Static). Это необходимо для возможности выноса отдельных модулей в микросервисы на других языках в будущем.
- **Связи в БД:** логические ссылки (без физических `FOREIGN KEY`), использование `UUID` в качестве первичных ключей.
- **Консистентность:** событийная консистентность вместо распределенных транзакций; денормализация данных для автономности контекстов.

### 4.2 Хранение изображений
- Все изображения (референсы и работы пользователей) хранятся на удаленном сервере (CDN).
- Конкретный CDN-провайдер на момент старта разработки **не выбран**; реализация должна быть абстрагирована (S3-compatible API, через Laravel Filesystem драйвер `s3` или аналог).
- Файлы загружаемых изображений: любые форматы изображений, максимальный размер **20 МБ**.
- Автоматическая генерация превью (thumbnails) **не требуется** в MVP; хранятся и отдаются оригиналы.

### 4.3 Очереди и фоновые задачи
- Для событийной консистентности и фоновой обработки используются Laravel Queues (драйвер Redis или Database).
- Примеры событий, обрабатываемых асинхронно:
  - `ArtworkLiked` — обновление денормализованного счетчика `likes_count`.
  - `ArtworkApproved` / `ArtworkRejected` — запись аудита в `moderation_actions`.
  - `ReferenceSetPublished` — обновление видимости набора в ленте.

### 4.4 Документирование API
- Весь публичный и административный API должен быть задокументирован с помощью Swagger / OpenAPI (пакет `l5-swagger`).
- Документация генерируется из PHPDoc-аннотаций в контроллерах.
- Swagger UI должен быть доступен по адресу `/api/documentation`.

## 5. Схема данных (кратко)

Основные сущности и контексты:

| Контекст | Сущности |
|---|---|
| **Identity** | `users` (UUID, nickname, role, auth_hash, recovery_code_hash) |
| **Content** | `reference_categories`, `reference_images`, `reference_sets`, `reference_set_items` |
| **Artworks** | `artworks` (UUID, user_id, reference_set_id, CDN-ссылки, caption, status, likes_count, soft delete) |
| **Engagement** | `likes`, `comments` (заготовка) |
| **Moderation** | `moderation_actions` (аудит действий админа) |
| **Static** | `project_info` (key-value для описания, донатов, соцсетей) |

Полная схема описана в файле `morf.dbml`.

## 6. Ограничения и допущения MVP

- **Комментарии:** заложены в схеме БД, но API не реализуется в первой версии.
- **Премодерация:** отсутствует; используется постмодерация.
- **Превью изображений:** не генерируются, отдаются оригиналы.
- **CDN:** реализация через абстрактный S3-совместимый драйвер, конкретный провайдер будет выбран позже.
- **Админ-панель:** используется готовое решение (Nova / Filament), не кастомная разработка с нуля.
